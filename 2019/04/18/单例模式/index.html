单例模式(Singleton)

思路：

对一个类，私有其构造类，不对外提供new，统一对外提供一个公共静态方法访问该变量。

【参考】https://my.oschina.net/mdxlcj/blog/1786451

【参考】https://www.cnblogs.com/qwangxiao/p/10152474.html

为什么要用单例？

单例模式能够保证一个类仅有唯一的实例，并提供一个全局访问点。

例：在线人数问题，可抽象为“计数器”的对象，需要用到计数器的地方，通过new一个计数器对象，实现计数，

​ 但是假如有多个用户同时登陆的话，通过计数器获取到的在线人数是相同的，基于此实现的+1操作，会存在

​ 数据与真实数据不一致。

​ 因此：把计数器设计为一个全局对象，所有人公用同一份数据，就可以避免类似问题，单例的一种应用。

总共五种：懒汉、饿汉、双重校验、枚举、静态内部类

  • 枚举:

    功能完整、使用简洁、无偿地提供了序列化机制、在面对复杂的序列化或者反射攻击时仍然可以绝对防止多次实例化

  • 懒汉:

    在调用到方法的时候才会创建

  • 饿汉:

    类被初始化时就已经在内存中创建对象

  • 双重校验锁:

    懒汉的优化，解决线程不安全，但性能提升不多。

  • 静态内部类:

    解决双重校验低性能问题，使用内部类来维护单例的实现，JVM内部的机制能够保证当一个类被加载的时候，这个类的加载过程是线程互斥的，这样当我们第一次调用getInstance的时候，JVM能够帮我们保证instance只被创建一次， 并且会保证把赋值给instance的内存初始化完毕，这样我们就不用担心上面的问题，同时该方法也只会在第一次调用的时候使用互斥机制，这样就解决了低性能问题。

【懒汉】
【双重校验】双重锁懒汉模式(DCL)

​ 由于jvm存在乱序执行功能，DCL也会出现线程不安全的情况: 使用volatile修饰符

​ instance= new LazyPerson();

​ 这个步骤，其实在jvm里面的执行分为三步：
　　1.在堆内存开辟内存空间。
　　2.在堆内存中实例化LazyPerson里面的各个参数。
　　3.把对象指向堆内存空间。
　　
　　由于jvm存在乱序执行功能，所以可能在2还没执行时就先执行了3，如果此时再被切换到线程B上，由于执行了3，INSTANCE 已经非空了，会被直接拿出来用，这样的话，就会出现异常。这个就是著名的DCL失效问题。
　　
　　不过在JDK1.5之后，官方也发现了这个问题，故而具体化了volatile，即在JDK1.6及以后，只要定义为private volatile static LazyPerson instance = null;可解决DCL失效问题。volatile确保instance 每次均在主内存中读取，但会牺牲一点效率。
